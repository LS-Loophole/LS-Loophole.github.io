UCMS v1.5.0 XSS cedit cname parameters

Vulnerability Type :

Cross Site Scripting(XSS)

Vulnerability Version :

1.5.0

Recurring environment:

  • Windows 10
  • PHP 5.4.5
  • Apache 2.4.23

Vulnerability Description AND recurrence:

There is a stored XSS vulnerability in \ucms\sadmin\cedit.php.

Vulnerability recurrence:

1.First create a vulnerability page and visit the following URL:
http://www.ucm1.5.0.com/ucms/index.php?do=sadmin. Click to add a column and enter ceshi in the column name to create a page

2.Visit the vulnerability URL:

http://www.ucm1.5.0.com/ucms/index.php?do=sadmin and click "column configuration"

3.The trigger point of the vulnerability is in the column name, enter XSS PayLoad'ceshi</h3><script>alert('cnvd')</script>' here

4.Save the column name and refresh again to trigger the storage XSS vulnerability

Vulnerability analysis

  1. First look at our XSS PayLoad "ceshi</h3><script>alert('cnvd')</script>". Because the front-end vulnerability point output is in the h3 tag, we need to use the </h3> tag to close

  1. There are two main points here, one is that there is no filtering when stored in the database, and the other is that the XSS filter function is not used when outputting
  1. First look at the stored in the database without filtering. Locate the storage file /ucms/index.php?do=sadmin_ceditpost where the front-end data is transferred to the back-end. In this statement, our data is directly written into the table without filtering

  1. The $cname in line 143 is a variable that stores XSS data, and we push it up here. Look for the process of copying and transforming this variable. Here you can see that there is only one $cname=dbstr($_POST['cname']); and there is no filtering on this parameter below, indicating that this is a direct conversion of $_POST['cname '] data is saved into the ucms_channel database!

  1. After reading the vulnerable stored procedure, look at the read triggering process again! The vulnerability is triggered on line 26 of the \ucms\sadmin\cedit.php page, where $link['cname'] outputs our vulnerability xss PayLoad.

  1. Looking up here, you can see that the $link array is queried from the ucms_channel database. And we just saw in step 4 that the program writes the payload of the storage vulnerability into the database without filtering.

  1. The $link['cname'] here corresponds to the vulnerability points previously stored in ceditpost.php. And there is no filtering here, just output xss PayLoad directly!
