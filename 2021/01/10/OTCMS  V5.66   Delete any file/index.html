OTCMS V5.66 Delete any file

Vulnerability Type :

Insecure Permissions

Vulnerability Version :

V5.66

Recurring environment:

  • Windows 10
  • PHP 5.4.5
  • Apache 2.4.23

Vulnerability Description AND recurrence:

Vulnerability introduction
Through this vulnerability, you can delete any file on the server, including deleting the installation file, resulting in the whole website being reinstalled!

Vulnerability location
CMS/admin/template_ deal.php Delfile() function in file

if (strpos(substr($filePath,3),'..') !== false){
        JS::AlertBackEnd('禁止查询路径中出现“..”字符串');
}

Vulnerability analysis

The main reason for this vulnerability is that the code filtering is not strict, and the vulnerability code is as follows

if (strpos(substr($filePath,3),'..') !== false){
        JS::AlertBackEnd('禁止查询路径中出现“..”字符串');
}


You can see that the substr() function is used here, starting from the third bit to the last bit

substr($filePath,3)

Definition and Usage 

The substr() function returns part of the string.

Note: if the start parameter is negative and length is less than or equal to start, then length is 0.

Then put the extracted string into the strpos () method to match the character “..”

(strpos(substr($filePath,3),'..'

As you can see, if there is “..” here, then loops is true. This causes if to set up, then output and end.

'prohibit'.. 'string in query path'

In the process of penetration, most of the files we use for directory traversal are files followed by “..”, as follows

config.php/../../admin.php

However, if the above method is used, it will be filtered by the code logic and cannot be executed

if (strpos(substr($filePath,3),'..') !== false){
        JS::AlertBackEnd('禁止查询路径中出现“..”字符串');
}

You can only use “..” in the first three characters if you want to bypass

../admin.php

In this way, you can bypass the filtering of strpos, but you can see that you can only skip one layer of directory on the Internet. At this time, the relative path of the current web page is very important. We know that the vulnerability file path is

admin/template_deal.php

It’s under the directory admin, so it’s easy to do. You can directly jump up one layer to return to the root directory, so that the scope of deletion becomes the whole CMS!

Vulnerability trigger process

Until there are loopholes here, it is to find out how to trigger! The current file is known as template_ deal.php , the vulnerability is in the delfile() function!

First of all, through reading the code, it is determined that most of the functions in this file are functions, and the switch selected function exists in the header.

Therefore, this file can directly access the transfer data, or be referenced to transfer data!

First, find the file name to determine whether the front-end code references this file!

template_deal.php

You can see that template.php Template exists in the front-end form code_ del.php And the variable Mudi in switch is passed

You can visit it directly here template.php Find out where this field is

Click Open template management is our vulnerability trigger page!

You can see that the red delete button at the back is roughly the deletion point. Here, grab a packet and delete one of them. Take a look at the packet at random. As you can see, this is the trigger point of the vulnerability!
